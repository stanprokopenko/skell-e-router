# Brainstorming Ideas Into Designs

Help turn ideas into fully formed designs and specs through natural collaborative dialogue.

Start by understanding the current project context, then ask questions one at a time to refine the idea. Once you understand what you're building, present the design and get user approval.

<HARD-GATE>
Do NOT invoke any implementation skill, write any code, scaffold any project, or take any implementation action until you have presented a design and the user has approved it. This applies to EVERY project regardless of perceived simplicity.
</HARD-GATE>

## Anti-Pattern: "This Is Too Simple To Need A Design"

Every project goes through this process. A todo list, a single-function utility, a config change — all of them. "Simple" projects are where unexamined assumptions cause the most wasted work. The design can be short (a few sentences for truly simple projects), but you MUST present it and get approval.

## Checklist

You MUST create a task for each of these items and complete them in order:

1. **Explore project context** — check files, docs, recent commits
2. **Offer visual companion** (if topic will involve visual questions) — this is its own message, not combined with a clarifying question. See the Visual Companion section below.
3. **Ask clarifying questions** — one at a time, understand purpose/constraints/success criteria
4. **Propose 2-3 approaches** — with trade-offs and your recommendation
5. **Present design** — in sections scaled to their complexity, get user approval after each section
6. **Write design doc** — save to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` and commit
7. **Spec self-review** — quick inline check for placeholders, contradictions, ambiguity, scope (see below)
8. **User reviews written spec** — ask user to review the spec file before proceeding
9. **Transition to implementation** — invoke writing-plans skill to create implementation plan

## The Process

**Understanding the idea:**

- Check out the current project state first (files, docs, recent commits)
- Before asking detailed questions, assess scope: if the request describes multiple independent subsystems (e.g., "build a platform with chat, file storage, billing, and analytics"), flag this immediately. Don't spend questions refining details of a project that needs to be decomposed first.
- If the project is too large for a single spec, help the user decompose into sub-projects: what are the independent pieces, how do they relate, what order should they be built? Then brainstorm the first sub-project through the normal design flow. Each sub-project gets its own spec → plan → implementation cycle.
- For appropriately-scoped projects, ask questions one at a time to refine the idea
- Prefer multiple choice questions when possible, but open-ended is fine too
- Only one question per message - if a topic needs more exploration, break it into multiple questions
- Focus on understanding: purpose, constraints, success criteria

**Exploring approaches:**

- Propose 2-3 different approaches with trade-offs
- Present options conversationally with your recommendation and reasoning
- Lead with your recommended option and explain why

**Presenting the design:**

- Once you believe you understand what you're building, present the design
- Scale each section to its complexity: a few sentences if straightforward, up to 200-300 words if nuanced
- Ask after each section whether it looks right so far
- Cover: architecture, components, data flow, error handling, testing
- Be ready to go back and clarify if something doesn't make sense

**Design for isolation and clarity:**

- Break the system into smaller units that each have one clear purpose, communicate through well-defined interfaces, and can be understood and tested independently
- For each unit, you should be able to answer: what does it do, how do you use it, and what does it depend on?
- Can someone understand what a unit does without reading its internals? Can you change the internals without breaking consumers? If not, the boundaries need work.
- Smaller, well-bounded units are also easier for you to work with - you reason better about code you can hold in context at once, and your edits are more reliable when files are focused. When a file grows large, that's often a signal that it's doing too much.

**Working in existing codebases:**

- Explore the current structure before proposing changes. Follow existing patterns.
- Where existing code has problems that affect the work (e.g., a file that's grown too large, unclear boundaries, tangled responsibilities), include targeted improvements as part of the design - the way a good developer improves code they're working in.
- Don't propose unrelated refactoring. Stay focused on what serves the current goal.

## After the Design

**Documentation:**

- Write the validated design (spec) to `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`
  - (User preferences for spec location override this default)
- Use elements-of-style:writing-clearly-and-concisely skill if available
- Commit the design document to git

**Spec Self-Review:**
After writing the spec document, look at it with fresh eyes:

1. **Placeholder scan:** Any "TBD", "TODO", incomplete sections, or vague requirements? Fix them.
2. **Internal consistency:** Do any sections contradict each other? Does the architecture match the feature descriptions?
3. **Scope check:** Is this focused enough for a single implementation plan, or does it need decomposition?
4. **Ambiguity check:** Could any requirement be interpreted two different ways? If so, pick one and make it explicit.

Fix any issues inline. No need to re-review — just fix and move on.

**User Review Gate:**
After the spec review loop passes, ask the user to review the written spec before proceeding:

> "Spec written and committed to `<path>`. Please review it and let me know if you want to make any changes before we start writing out the implementation plan."

Wait for the user's response. If they request changes, make them and re-run the spec review loop. Only proceed once the user approves.

**Implementation:**

- Invoke the writing-plans skill to create a detailed implementation plan
- Do NOT invoke any other skill. writing-plans is the next step.

## Key Principles

- **One question at a time** - Don't overwhelm with multiple questions
- **Multiple choice preferred** - Easier to answer than open-ended when possible
- **YAGNI ruthlessly** - Remove unnecessary features from all designs
- **Explore alternatives** - Always propose 2-3 approaches before settling
- **Incremental validation** - Present design, get approval before moving on
- **Be flexible** - Go back and clarify when something doesn't make sense

## Visual Companion

A browser-based companion for showing mockups, diagrams, and visual options during brainstorming. Available as a tool — not a mode. Accepting the companion means it's available for questions that benefit from visual treatment; it does NOT mean every question goes through the browser.

**Offering the companion:** When you anticipate that upcoming questions will involve visual content (mockups, layouts, diagrams), offer it once for consent:
> "Some of what we're working on might be easier to explain if I can show it to you in a web browser. I can put together mockups, diagrams, comparisons, and other visuals as we go. This feature is still new and can be token-intensive. Want to try it? (Requires opening a local URL)"

**This offer MUST be its own message.** Do not combine it with clarifying questions, context summaries, or any other content. The message should contain ONLY the offer above and nothing else. Wait for the user's response before continuing. If they decline, proceed with text-only brainstorming.

**Per-question decision:** Even after the user accepts, decide FOR EACH QUESTION whether to use the browser or the terminal. The test: **would the user understand this better by seeing it than reading it?**

- **Use the browser** for content that IS visual — mockups, wireframes, layout comparisons, architecture diagrams, side-by-side visual designs
- **Use the terminal** for content that is text — requirements questions, conceptual choices, tradeoff lists, A/B/C/D text options, scope decisions

A question about a UI topic is not automatically a visual question. "What does personality mean in this context?" is a conceptual question — use the terminal. "Which wizard layout works better?" is a visual question — use the browser.

If they agree to the companion, read the detailed guide before proceeding:
`skills/brainstorming/visual-companion.md`

70	def _encode_image(source: str) -> dict:
71	    """Convert an image source (URL, data URI, or file path) to an OpenAI-format content part."""
72	    if source.startswith(("http://", "https://")):
73	        return {"type": "image_url", "image_url": {"url": source}}
74	    if source.startswith("data:"):
75	        return {"type": "image_url", "image_url": {"url": source}}
76	    # File path -- read, detect MIME, base64 encode
77	    if not os.path.isfile(source):
78	        raise RouterError(
79	            code="INVALID_INPUT",
80	            message=f"Image file not found: {source}"
81	        )
82	    mime_type, _ = mimetypes.guess_type(source)
83	    if mime_type is None:
84	        mime_type = "application/octet-stream"
85	    with open(source, "rb") as f:
86	        encoded = base64.b64encode(f.read()).decode("utf-8")
87	    return {"type": "image_url", "image_url": {"url": f"data:{mime_type};base64,{encoded}"}}

90	def upload_file(
91	    file: bytes | BinaryIO,
92	    mime_type: str,
93	    display_name: str | None = None,
94	    config: dict | None = None,
95	) -> GeminiFileRef:
96	    """Upload a file to Gemini's Files API and return a GeminiFileRef.
97	
98	    Uses litellm.create_file under the hood.  The returned ref can be passed
99	    to ask_ai via the ``files`` parameter.
100	    """
101	    # Resolve API key: config dict first, then env var
102	    config = config or {}
103	    api_key = config.get("gemini_api_key") or os.environ.get("GEMINI_API_KEY")
104	    if not api_key:
105	        raise RouterError(
106	            code="MISSING_ENV",
107	            message="GEMINI_API_KEY is required for file uploads.",
108	            details={"required": "GEMINI_API_KEY"},
109	        )
110	
111	    # Read bytes from BinaryIO if needed
112	    file_content = file if isinstance(file, bytes) else file.read()
113	
114	    filename = display_name or "upload"
115	    try:
116	        result = litellm.create_file(
117	            file=(filename, file_content, mime_type),
118	            purpose="assistants",
119	            custom_llm_provider="gemini",
120	            api_key=api_key,
121	        )
122	    except Exception as e:
123	        safe_msg = _redact_keys(str(e), config)
124	        raise RouterError(
125	            code="UPLOAD_ERROR",
126	            message=safe_msg,
127	            details={"mime_type": mime_type},
128	        ) from e
129	
130	    return GeminiFileRef(uri=result.id, mime_type=mime_type, display_name=display_name)

134	def _construct_messages(user_input: str | list[dict], system_message: str = None, images: list[str] | None = None, files: list[GeminiFileRef] | None = None):
135	    messages = []
136	
137	    if images and not isinstance(user_input, str):
138	        raise RouterError(
139	            code="INVALID_INPUT",
140	            message="'images' parameter is only supported when 'user_input' is a string. "
141	                    "For list input, embed image content parts directly in your messages."
142	        )
143	
144	    if files and not isinstance(user_input, str):
145	        raise RouterError(
146	            code="INVALID_INPUT",
147	            message="'files' parameter is only supported when 'user_input' is a string. "
148	                    "For list input, embed file content parts directly in your messages."
149	        )
150	
151	    if system_message:
152	        messages.append({"role": "system", "content": system_message})
153	
154	    if isinstance(user_input, str):
155	        if images or files:
156	            content_parts = [{"type": "text", "text": user_input}]
157	            for img in (images or []):
158	                content_parts.append(_encode_image(img))
159	            for ref in (files or []):
160	                content_parts.append({"type": "file", "file": {"file_id": ref.uri, "format": ref.mime_type}})
161	            messages.append({"role": "user", "content": content_parts})
162	        else:
163	            messages.append({"role": "user", "content": user_input})
164	    elif isinstance(user_input, list):
165	        # Ensure all items in the list are dictionaries with 'role' and 'content'
166	        if all(isinstance(msg, dict) and 'role' in msg and 'content' in msg for msg in user_input):
167	            messages.extend(user_input)
168	        else:
169	            raise RouterError(
170	                code="INVALID_INPUT",
171	                message="user_input items must be dicts with 'role' and 'content'"
172	            )
173	    else:
174	        raise RouterError(
175	            code="INVALID_INPUT",
176	            message=f"user_input must be a string or list of dicts with 'role' and 'content', got {type(user_input)}"
177	        )
178	    return messages

201	# Classify retryable errors and honor Retry-After when available
202	def _extract_status_and_headers(exc: Exception) -> tuple[int | None, dict]:
203	    status = getattr(exc, 'status_code', None)
204	    headers = getattr(exc, 'headers', None)
205	    if headers is None:
206	        resp = getattr(exc, 'response', None)
207	        if resp is not None:
208	            status = getattr(resp, 'status_code', status)
209	            headers = getattr(resp, 'headers', None)
210	    if headers is None:
211	        headers = {}
212	    return status, headers

215	def _parse_retry_after_seconds(headers: dict) -> float | None:
216	    if not isinstance(headers, dict):
217	        return None
218	    retry_after = None
219	    for k, v in headers.items():
220	        if str(k).lower() == 'retry-after':
221	            retry_after = v
222	            break
223	    if retry_after is None:
224	        return None
225	    try:
226	        seconds = float(str(retry_after).strip())
227	        if seconds >= 0:
228	            return seconds
229	    except Exception:
230	        return None
231	    return None

234	def _is_quota_related(exc: Exception) -> bool:
235	    code = getattr(exc, 'code', None)
236	    if code is None:
237	        error = getattr(exc, 'error', None)
238	        if isinstance(error, dict):
239	            code = error.get('code') or error.get('type')
240	    if isinstance(code, str):
241	        c = code.lower()
242	        return ('quota' in c) or ('insufficient_quota' in c) or ('quota_exceeded' in c)
243	    return False

246	def _is_retryable_exception(exc: Exception) -> bool:
247	    status, headers = _extract_status_and_headers(exc)
248	
249	    name = exc.__class__.__name__.lower()
250	    if 'timeout' in name or 'connection' in name or 'connect' in name:
251	        return True
252	
253	    if status is None:
254	        return False
255	
256	    if status in (500, 502, 503, 504):
257	        # Optional: respect very large Retry-After on 503 if present
258	        ra = _parse_retry_after_seconds(headers)
259	        if ra is not None and ra > 120:
260	            return False
261	        return True
262	
263	    if status == 429:
264	        if _is_quota_related(exc):
265	            return False
266	        ra = _parse_retry_after_seconds(headers)
267	        if ra is not None and ra > 120:
268	            return False
269	        return True
270	
271	    return False

274	def _retry_after_wait(retry_state) -> float:
275	    exc = retry_state.outcome.exception() if retry_state.outcome else None
276	    if exc is None:
277	        return FALLBACK_WAIT(retry_state)
278	
279	    status, headers = _extract_status_and_headers(exc)
280	    if status in (429, 503):
281	        seconds = _parse_retry_after_seconds(headers)
282	        if seconds is not None and seconds >= 0:
283	            return min(seconds, 120.0)
284	        return FALLBACK_WAIT(retry_state)
285	
286	    return FALLBACK_WAIT(retry_state)

## Gemini Multimodal Embeddings in LiteLLM

LiteLLM supports `gemini-embedding-2-preview` which provides multimodal embeddings for text, images, audio, video, and PDF in a single request.

### How Aggregation Works

The key distinction between different routing paths affects how inputs are aggregated:

**Gemini API Path:**
LiteLLM routes to Gemini's batchEmbedContents endpoint with one EmbedContentRequest per input, which differs from the Vertex AI path that combines all parts into a single unified vector.

**Vertex AI Path:**
Vertex AI's Gemini embedding endpoint only exposes single-content embedContent (no batchEmbedContents), so passing N items in input=[...] returns 1 unified embedding that fuses all parts.

### Creating Aggregated Embeddings

Submitting multiple parts (for example, text and an image) within a single content entry produces one aggregated embedding for all modalities within that entry.

### Supported Input Format

Supported MIME types include image/png, image/jpeg, audio/mpeg, audio/wav, video/mp4, video/quicktime, and application/pdf.

The implementation allows you to pass multimodal inputs using base64-encoded data URIs or GCS URLs for different media types, all processed through LiteLLM's unified embedding interface.

```python
def resolve_embedding_alias(alias: str) -> EmbeddingModel:
    model = EMBEDDING_MODEL_CONFIG.get(alias)
    if not model:
        raise RouterError(
            code="INVALID_MODEL",
            message=f"Invalid embedding model alias '{alias}'. "
                    f"Available: {sorted(k for k in EMBEDDING_MODEL_CONFIG if '/' not in k)}"
```

**Input handling pipeline (in order):**

1. **Resolve model** — `resolve_embedding_alias(model)` → `EmbeddingModel`. Raises `RouterError("INVALID_MODEL")` if unknown.
2. **Provider key check** — `_check_provider_key(...)` (reused from chat path).
3. **Normalize input shape** — internal helper `_normalize_input(input, embedding_model)`:
   - `str` → wrap as `[input]` for LiteLLM, remember `was_str=True` (for unwrapping return).
   - Each top-level element is one of: `str`, `GeminiFileRef`, `list[str | GeminiFileRef]` (nested = aggregate), or a local file path (string starting with `./`, `/`, drive letter, or existing file).
   - For each `str` element: detect modality (text vs file path) via `_classify_input_part`. File paths get `_encode_to_data_uri` (reuses `_encode_image` logic, extended to other MIME types).
   - For each `GeminiFileRef`: convert to LiteLLM file reference dict.
   - Validate inferred modalities against `embedding_model.supported_inputs` — raise `RouterError("INVALID_INPUT", "<model> does not support <modality>")`.
   - Validate aggregation — if any top-level element is a `list`, require `embedding_model.supports_aggregation` — else `RouterError("INVALID_INPUT", "...does not support aggregation; flatten the list for batch")`.
4. **Validate `dimensions`** — if set, must be `≤ embedding_model.max_dimensions` and `≥ 1`. Else `RouterError("INVALID_PARAM")`. Pass-through to LiteLLM (LiteLLM maps to `dimensions` for OpenAI / `output_dimensionality` for Gemini).
5. **Call LiteLLM with retry** — see below.
6. **Build response** — extract `data[*].embedding` in index order, unwrap if `was_str`, optionally wrap in `EmbeddingResponse`.

**`_classify_input_part(part: str)` — detection rules:**

| Pattern | Inferred modality |
|---|---|
| `data:image/...` | image |
| `data:audio/...` | audio |
| `data:video/...` | video |
| `data:application/pdf` | pdf |
| `https://...` or `http://...` ending in known image/audio/video/pdf MIME extension | by extension |
| `gs://...` | passes through (Vertex GCS — modality from extension) |
| Existing file path (`os.path.isfile`) | by `mimetypes.guess_type` → image/audio/video/pdf/text |
| Anything else | text |

Failure path: if a string looks like a file path (e.g., starts with `./` or `/`) but the file doesn't exist → `RouterError("INVALID_INPUT", "File not found: <path>")`. Plain text is never misclassified because it only matches text after no other pattern hits.

**Retry / backoff:**

```python
@retry(

def _perform_embedding(model_name: str, input: list, api_key: str | None, **kwargs):

    if api_key:
```

```python
def _build_embedding_response(
    response, embedding_model, request_duration_s, total_duration_s
) -> EmbeddingResponse:
    # data is sorted by index already; defensive sort just in case
    data = sorted(response.data, key=lambda d: d["index"] if isinstance(d, dict) else d.index)
    embeddings = [

        for d in data
```

137	### `EmbeddingModel`
138	
139	```python
140	class EmbeddingModel:
141	    def __init__(
142	        self,
143	        name: str,                              # full LiteLLM model string
144	        provider: str,                          # "openai" | "gemini"
145	        supported_inputs: set[str],             # subset of {"text","image","audio","video","pdf"}
146	        max_dimensions: int,
147	        default_dimensions: int,
148	        recommended_dimensions: tuple[int, ...] = (),
149	        max_input_tokens: int | None = None,
150	        supports_aggregation: bool = False,
151	        native_batch: bool = True,
152	    ): ...
153	
154	    @property
155	    def is_openai(self) -> bool: return self.provider == "openai"
156	    @property
157	    def is_gemini(self) -> bool: return self.provider == "gemini"
158	```
159

162	### `EmbeddingResponse`
163	
164	```python
165	from dataclasses import dataclass
166	from typing import Any
167	
168	@dataclass
169	class EmbeddingResponse:
170	    embeddings: list[list[float]]      # always nested, even for single-input
171	    model: str                          # provider's reported model name
172	    dimensions: int                     # observed: len(embeddings[0])
173	
174	    prompt_tokens: int | None = None
175	    total_tokens: int | None = None
176	
177	    cost: float | None = None
178	    duration_seconds: float | None = None
179	    total_duration_seconds: float | None = None
180	
181	    raw_response: Any = None

184	## Execution flow
185	
186	1. **Resolve model** — `resolve_embedding_alias(model)` → `EmbeddingModel`.
187	2. **Validate verbosity** (same lower-case check as `ask_ai`).
188	3. **Check provider key** — `_check_provider_key(...)`; raises `MISSING_ENV`.
189	4. **Normalize input**:
190	   - Track `was_str = isinstance(input, str)`; if so, wrap as `[input]`.
191	   - Walk the (now always-list) input. For each top-level element:
192	     - If it's a string or `GeminiFileRef`, classify modality and convert to a single LiteLLM part.
193	     - If it's a `list`, mark aggregation; classify each part; produce a nested LiteLLM list.
194	   - Validate inferred modality set ⊆ `embedding_model.supported_inputs`.
195	   - If any aggregation occurred, require `embedding_model.supports_aggregation`.
196	5. **Validate `dimensions`** — `1 ≤ dimensions ≤ max_dimensions` if provided.
197	6. **Resolve API key** — `_resolve_api_key(embedding_model, config)`; falls through to LiteLLM's env-var lookup if `None`.
198	7. **Call LiteLLM with retry**:
199	   ```python
200	   @retry(
201	       retry=retry_if_exception(_is_retryable_exception),
202	       wait=_retry_after_wait,
203	       stop=stop_after_attempt(3),
204	   )
205	   def _perform_embedding(model_name, input, api_key=None, **kwargs):
206	       request_start = time.perf_counter()
207	       response = litellm.embedding(
208	           model=model_name, input=input,
209	           **({"api_key": api_key} if api_key else {}),
210	           **kwargs,
211	       )
212	       return response, time.perf_counter() - request_start
213	   ```
214	   `kwargs` is `{"dimensions": dimensions}` if set, else empty.
215	8. **Wrap LiteLLM exceptions** in `RouterError("PROVIDER_ERROR", _redact_keys(str(e), config), ...)`.
216	9. **Build response** — extract `data[*].embedding` sorted by index; compute `len(embeddings[0])`; compute cost via `litellm.completion_cost(...)` (best-effort, swallow exceptions like the chat path does).
217	10. **Return**:
218	    - `rich_response=True` → `EmbeddingResponse`.
219	    - `rich_response=False` and `was_str=True` → `embeddings[0]` as `list[float]`.
220	    - `rich_response=False` and `was_str=False` → `embeddings` as `list[list[float]]`.
221	
222	## Verbosity behavior
223	
224	Mirrors `ask_ai`'s policy but skips the "RESPONSE" content block (vectors are noise to print):
225	
226	- `none` — silent.
227	- `response` — prints `ASKING AI (<model>)…` only.
228	- `info` — also prints model, dimension count, prompt tokens, total tokens, cost, duration.
229	- `debug` — also pretty-prints kwargs and the raw response.

231	## Testing
232	
233	`tests/test_embeddings.py` mocks `litellm.embedding` via the same patching pattern used in `tests/test_utils.py`. Test groups:
234	
235	| Group | Coverage |
236	|---|---|
237	| Alias resolution | unknown alias raises `INVALID_MODEL`; full-name lookup works; both short and full names resolve to the same `EmbeddingModel` |
238	| Single-input normalization | `str` input becomes `[str]` going in; return is unwrapped `list[float]` |
239	| Batch-input normalization | flat `list[str]` passes through unchanged; return is `list[list[float]]` |
240	| Aggregation | nested list flagged as aggregate; rejected on OpenAI with `INVALID_INPUT`; allowed on Gemini |
241	| File-path encoding | local image path → data URI; missing file → `INVALID_INPUT` |
242	| Modality classification | data URI / file path / URL / plain text classifications correct |
243	| Capability validation | OpenAI + image → `INVALID_INPUT`; Gemini + audio file → ok; Gemini + nested → ok |
244	| Dimension validation | `dimensions=4096` on small → `INVALID_PARAM`; in-range value passes through to LiteLLM kwargs |
245	| Retry behavior | 503 retries; 429-with-quota does not retry; 200 does not retry; respects `Retry-After` |
246	| Response building | `EmbeddingResponse` fields populated; `dimensions = len(embeddings[0])`; `cost=None` when `litellm.completion_cost` raises; index ordering preserved |

# Writing Plans

## Overview

Write comprehensive implementation plans assuming the engineer has zero context for our codebase and questionable taste. Document everything they need to know: which files to touch for each task, code, testing, docs they might need to check, how to test it. Give them the whole plan as bite-sized tasks. DRY. YAGNI. TDD. Frequent commits.

Assume they are a skilled developer, but know almost nothing about our toolset or problem domain. Assume they don't know good test design very well.

**Announce at start:** "I'm using the writing-plans skill to create the implementation plan."

**Context:** This should be run in a dedicated worktree (created by brainstorming skill).

**Save plans to:** `docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md`
- (User preferences for plan location override this default)

## Scope Check

If the spec covers multiple independent subsystems, it should have been broken into sub-project specs during brainstorming. If it wasn't, suggest breaking this into separate plans — one per subsystem. Each plan should produce working, testable software on its own.

## File Structure

Before defining tasks, map out which files will be created or modified and what each one is responsible for. This is where decomposition decisions get locked in.

- Design units with clear boundaries and well-defined interfaces. Each file should have one clear responsibility.
- You reason best about code you can hold in context at once, and your edits are more reliable when files are focused. Prefer smaller, focused files over large ones that do too much.
- Files that change together should live together. Split by responsibility, not by technical layer.
- In existing codebases, follow established patterns. If the codebase uses large files, don't unilaterally restructure - but if a file you're modifying has grown unwieldy, including a split in the plan is reasonable.

This structure informs the task decomposition. Each task should produce self-contained changes that make sense independently.

## Bite-Sized Task Granularity

**Each step is one action (2-5 minutes):**
- "Write the failing test" - step
- "Run it to make sure it fails" - step
- "Implement the minimal code to make the test pass" - step
- "Run the tests and make sure they pass" - step
- "Commit" - step

## No Placeholders

Every step must contain the actual content an engineer needs. These are **plan failures** — never write them:
- "TBD", "TODO", "implement later", "fill in details"
- "Add appropriate error handling" / "add validation" / "handle edge cases"
- "Write tests for the above" (without actual test code)
- "Similar to Task N" (repeat the code — the engineer may be reading tasks out of order)
- Steps that describe what to do without showing how (code blocks required for code steps)
- References to types, functions, or methods not defined in any task

- Exact file paths always
- Complete code in every step — if a step changes code, show the code
- Exact commands with expected output

# Export Claude Conversation

Convert raw Claude Code session `.jsonl` files into human-readable `.txt` transcripts. The bundled script is deterministic — same input always produces the same output.

## When to use

Trigger any time the user wants conversation history outside the raw jsonl form: archiving, sharing with a teammate, feeding into another LLM, pasting into a doc, or just reading. If the user mentions "the conversation" / "this chat" / "session", and they want it in a file, this skill applies.

## Output location and filenames

Always save to `<current-project-cwd>/docs/ai-conversations/`. Create the directory if it doesn't exist. Never write outside this directory.

For each session you export, produce **both** files unless the user asks for only one:

- `<filename>.txt` — full transcript (USER, ASSISTANT, TOOL RESULT, TEAMMATE, SYSTEM REMINDER, SKILL CONTENT, etc.)
- `<filename>-clean.txt` — only USER and ASSISTANT text

If the skill is invoked with **no qualifying context** — e.g. the user just types `/export-claude-conversation` on its own, or says something like "export the conversation" without specifying which one — default to exporting the **current** session (see "Current session detection" below). Pick a filename, run the export, don't ask first.

### Filename convention

`<YYYY-MM-DD>-<short-kebab-description>.txt`

- Date = the session's first-event timestamp.
- Description = 2–6 kebab-case words capturing the topic. Read the first user message of the session to choose. If the user provides a name, use theirs verbatim.
- Don't include the session-id UUID. Filenames are for humans.

When exporting multiple sessions on the same topic across consecutive days, suffix with `-attempt1` / `-continued` / `-followup` so order is obvious.

## Finding sessions

Claude Code stores one `.jsonl` per session under `~/.claude/projects/<project-slug>/<session-id>.jsonl`. The slug is the project's absolute path with separators rewritten:

- Windows: `C:\Users\Stan\Documents\GitHub\foo` → `C--Users-Stan-Documents-GitHub-foo`
- POSIX: `/Users/stan/code/foo` → `-Users-stan-code-foo`

If the deterministic slug doesn't match a real directory, list `~/.claude/projects/` and pick the entry whose name resembles the current cwd.

### Filtering

- **By recency** ("past N days", "last N"): use file mtime. Larger files generally mean longer sessions, not necessarily more recent.
- **By topic** ("conversations about X"): grep the jsonl for the keyword. Then read each match's first user message to confirm the topic — keyword presence alone is noisy.
- **By session ID**: if the user provides one, use it directly.

### Current session detection

There's no first-class "current session" pointer. Heuristic: the jsonl with the **most recent mtime** in the project's session directory is virtually always the current session (it's being appended to as the conversation runs). Use that.

## Running the export script

The script lives at `<this-skill>/scripts/export_jsonl_conversation.py`. Invoke with `py` on Windows or `python3` on POSIX.

```bash
py "<skill-path>/scripts/export_jsonl_conversation.py" \
   "<session-jsonl-path>" \
   "<cwd>/docs/ai-conversations/<filename>.txt"

py "<skill-path>/scripts/export_jsonl_conversation.py" --clean \
   "<session-jsonl-path>" \
   "<cwd>/docs/ai-conversations/<filename>-clean.txt"
```

Useful flags:

- `--clean` — emit only USER and ASSISTANT text, stripping tool calls/results, thinking blocks, system reminders, skill auto-loads, and subagent (`<teammate-message>`) reports.
- `--max-tool-chars N` — cap any single tool result at N characters in the full export. Default 20000. Pass `0` to disable truncation. Only relevant for the full export.

When exporting many sessions in one go, run them as separate invocations (no batch mode is needed; the script is fast).

## What the full export labels mean

The renderer distinguishes:

- `USER` — actual human input
- `ASSISTANT` — assistant turns (text + tool calls + thinking)
- `TOOL RESULT` — tool output delivered back to the assistant
- `TEAMMATE` / `TEAMMATE EVENT` / `TASK STATUS` — subagent / team-workflow events
- `SYSTEM` / `SYSTEM REMINDER` — harness-injected reminders
- `SKILL CONTENT` / `SKILL LAUNCH` — skill auto-loading payloads
- `LOCAL COMMAND OUTPUT` / `LOCAL COMMAND ERROR` / `LOCAL COMMAND CAVEAT` — slash-command stdout/stderr
- `USER INTERRUPT` — `[Request interrupted by user]` markers

These distinctions matter because in the raw jsonl the API delivers many of these as `user`-role messages — naive readers would mistake subagent reports and skill content for human input. The clean export drops everything except real `USER` and `ASSISTANT` text.

## Workflow

1. **Resolve the project's session directory** (deterministic slug → fallback to listing `~/.claude/projects/`).
2. **Pick the target session(s)** based on the user's request: current (latest mtime), past N days (mtime filter), topical (grep + first-message confirmation), or explicit session ID.
3. **Propose filenames** to the user before exporting more than one or two — they'll often have better names. For a single current-session export with no specifics, just pick a sensible name and go.
4. **Create `docs/ai-conversations/`** in the cwd if it doesn't exist.
5. **Run the script twice per session** (full + clean) unless the user asked for one.
6. **Report back** with the file paths and one-line summary of what each session is about.

## Tips

- For "export the last 5 conversations" or similar, list candidates briefly to the user with first-user-message snippets before proposing filenames — gives them a chance to drop irrelevant ones.
- The script is the source of truth for label logic. If the user reports a mislabeled message, look at the script's `_USER_TEXT_LABEL_MAP` and `_SYSTEM_TAG_RE` — those are the only two places labeling/filtering decisions are made.
- The clean export is much smaller (typically 5–20% of the full size). When the user asks for "the conversation" without qualifying, the clean version is almost always what they want to actually read; the full version is for diagnostics.
